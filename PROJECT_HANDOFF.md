# Franchise Insight Agent Handoff

이 문서는 Mac에서 작업한 내용을 Windows 컴퓨터나 다음 Codex 세션에서 바로 이어가기 위한 인수인계 메모입니다.

## 현재 목적

이 앱은 단순 키워드 조회기가 아니라, 프랜차이즈 양도양수 글감을 고르기 위한 인사이트 보드입니다.

핵심 질문은 아래입니다.

- 오늘 어떤 브랜드/지역 조합을 먼저 봐야 하는가?
- 단순 검색량 1등과, 최근 흐름이 평소와 달라진 이상 신호는 무엇인가?
- 선택한 브랜드가 지역별로 어디에서 반응이 있는가?
- 이 결과를 글쓰기 에이전트에 어떤 제목/방향으로 넘길 것인가?

## 현재까지 정리한 방향

### 1. 필터는 검색창형으로 변경

이전에는 브랜드와 지역을 여러 개 칩으로 펼쳐 보여줬지만, 브랜드가 많아질수록 눈에 안 들어오고 필터 역할도 약했습니다.

현재 방향은 아래입니다.

- 브랜드 입력: 예) `메가커피`
- 지역 입력: 예) `서울 노원구`
- `분석하기` 버튼 또는 Enter
- 입력값이 없으면 시장 전체 기준으로 분석
- 브랜드만 입력하면 해당 브랜드의 지역별 흐름 분석
- 지역만 입력하면 해당 지역 안의 브랜드별 흐름 분석
- 브랜드+지역을 같이 입력하면 해당 조합 중심 분석

### 2. 검색량 상위와 이상 신호는 둘 다 유지

사용자 피드백상, 검색량 1등만 보거나 이상 신호만 보는 것은 부족합니다.

현재 화면은 `오늘 볼 신호`를 두 칸으로 나눕니다.

- 검색량 상위: 월검색량 자체가 큰 후보
- 이상 신호: 30일 평균 대비 최근 7일 흐름이 달라진 후보

둘 중 하나를 없애지 말고, 서로 다른 판단 근거로 유지해야 합니다.

### 3. 브랜드 수요 순위는 기본 시장 목록 기준

기존 기본 브랜드가 8개뿐이라 `TOP 10`이라고 해도 8개만 보였습니다.

현재는 기본 브랜드 목록을 24개로 늘렸고, 화면 제목도 실제 개수에 맞게 보이도록 정리했습니다.

예: `브랜드 수요 순위 (24개 중 상위 10개)`

### 4. 캐시 fallback 제거

중요한 수정입니다.

예전에는 조건에 맞는 캐시가 없으면 최신 캐시를 대신 보여줬습니다. 검색창형 필터에서는 이것이 위험합니다.

예를 들어 `메가커피 / 서울 노원구`를 입력했는데, 예전 전체 시장 캐시가 섞여 나오면 사용자가 필터가 안 먹는다고 느낍니다.

현재는 입력 조건과 정확히 맞는 캐시만 사용하고, 없으면 새로 수집합니다.

## 다음 작업 우선순위

### 1. 그래프 해석 개선

현재 그래프는 선만 보여줘서 변화 폭이 직관적으로 느껴지지 않습니다.

다음 개선안:

- 30일 평균 기준선 표시
- 최근 7일 구간 강조
- 마지막 값 라벨 표시
- 7일 평균, 30일 평균, 변화율을 카드로 분리
- `+25` 같은 숫자만 보여주지 말고 `30일 평균 대비 최근 7일 +18%`처럼 해석 문구 제공

### 2. 직접 검색 결과 상세 화면 개선

브랜드+지역을 입력했을 때 아래 정보가 한눈에 보여야 합니다.

- 핵심 키워드
- 월검색량
- 7일/30일 흐름
- 블로그 경쟁
- 왜 이 조합을 봐야 하는지
- 글쓰기 에이전트에 넘길 제목/방향

현재는 구조가 잡혔고, 설명 밀도와 시각화는 더 다듬어야 합니다.

### 3. 추가 키워드 결과를 별도 섹션으로 분리

현재 추가 키워드는 새 수집 때 분석에 포함된다는 상태 문구만 보입니다.

다음 개선안:

- `직접 입력 키워드` 섹션 추가
- 입력한 키워드별 검색량/추이/블로그 경쟁 표시
- 브랜드/지역 자동 추출이 애매한 키워드도 결과에서 확인 가능하게 만들기

### 4. 블로그 경쟁 판단 고도화

네이버 공식 API만으로는 블로그 지수, 실제 클릭 수, 구독자 수를 직접 가져오기 어렵습니다.

대신 가능한 대체 지표:

- 블로그 문서 총량
- 최근 글 비중
- 제목에 광고성 문구가 많은지
- 같은 블로그나 비슷한 제목 반복 여부
- 검색량 대비 문서량이 과한지

목표는 `블로그 경쟁 높음` 한 줄이 아니라 아래처럼 나누는 것입니다.

- 문서량 과다
- 광고성 제목 많음
- 검색량 대비 경쟁 낮음
- 제목 표본 부족

### 5. API 사용량 표시

여러 명이 쓸 수 있으므로 `새로 수집` 전에 예상 호출량을 보여주는 게 좋습니다.

다음 개선안:

- 이번 수집 예상 키워드 수
- SearchAd API 예상 호출 수
- Open API 예상 호출 수
- 현재 캐시 사용 여부
- 오늘 몇 시 이후 캐시인지

### 6. 검색 실행 방식과 버튼 UX 정리

현재 분석 조건에서 브랜드/지역 입력칸은 확인했습니다.

- `분석하기` 버튼을 눌러도 검색됨
- 입력칸에서 Enter를 직접 눌러도 검색됨

다음 세션에서 고민할 것:

- 버튼과 Enter를 둘 다 유지하는 것이 사용자에게 맞는지 확인
- 둘 다 유지한다면 화면 문구로 `Enter 또는 분석하기`를 분명히 안내
- Enter 검색이 실수로 새 수집을 많이 유발하지 않는지 확인
- `분석하기`는 저장 데이터/캐시 기준인지, 새 API 수집까지 하는지 명확히 구분

현재 버튼들에 대한 고민:

- `저장 데이터 보기`를 눌렀을 때 화면 변화가 거의 없어 보임
- `연결 상태 확인`이 실제 사용자에게 어떤 의미인지 직관적이지 않음
- `오늘 데이터 새로 수집`은 API 호출이 발생할 수 있으므로 더 조심스럽게 보여줘야 함

다음 개선 방향 후보:

- 기본 행동: 브랜드/지역 입력 후 Enter 또는 `분석하기`
- 캐시 사용 여부를 화면에 표시
- API를 새로 쓰는 행동은 `새로 수집`으로만 분리
- `저장 데이터 보기`는 제거하거나 이름을 `저장된 결과 다시 보기`로 변경
- `연결 상태 확인`은 별도 버튼보다 시작 점검 영역에서 자동으로 보이게 하기
- 버튼 옆에 `API 사용 없음`, `API 사용 가능성 있음` 같은 작은 안내를 붙일지 검토

내일 시작하면 이 버튼 UX부터 먼저 정리하는 것이 좋습니다. 지금 기능은 연결되어 있지만, 사용자가 언제 어떤 버튼을 눌러야 하는지 아직 충분히 친화적이지 않습니다.

### 7. Windows 실행 확인

Windows 컴퓨터에서 이어받을 때 확인할 것:

- GitHub Desktop에서 Pull 후 저장소 루트가 `franchise_insight_agent`인지 확인
- `keyword-insight-agent/start-keyword-insight-agent.bat` 실행
- 브라우저에서 표시된 localhost 주소 접속
- `naver-api-config.json`은 GitHub에 올라가지 않으므로 Windows에도 별도로 준비 필요
- `naver-api-config.example.json`은 예시 파일일 뿐 실제 키가 아님
